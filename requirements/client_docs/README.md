# Client Documentation

Client-provided documentation and materials.

## Purpose

Store all documentation provided by the client including existing system docs, business processes, and technical specifications.

## Content Types

- Existing system documentation
- Business process documents
- Technical specifications from client
- API documentation
- Brand guidelines and style guides
- User manuals
- Training materials
- Architectural diagrams

## Naming Convention

- Keep original filenames when possible
- Add date prefix if multiple versions: `YYYYMMDD_[ORIGINAL_FILENAME]`
- Organize into subdirectories by category if needed

## Organization

Consider creating subdirectories for large projects:
```
client_docs/
├── existing_systems/
├── business_processes/
├── technical_specs/
├── brand_guidelines/
└── training_materials/
```

## Best Practices

1. **Preserve Originals**: Keep files in original format
2. **Version Tracking**: Track version numbers if client provides updates
3. **Cataloging**: Maintain index of all client docs in this README
4. **Access**: Respect client confidentiality requirements
5. **Conversion**: Consider converting PDFs to Markdown for easier version control
6. **Validation**: Verify documents are current and accurate with client
